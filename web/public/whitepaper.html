§ 1

Abstract

HumbleTrust is a trust-layer protocol built on Solana that makes token launches verifiably honest — before the first trade happens.

The Solana ecosystem has unprecedented velocity but a structural trust problem: creators can promise anything in a whitepaper yet retain full control of liquidity and supply. HumbleTrust changes the rules at the infrastructure level. Instead of asking for trust, the protocol enforces it on-chain.

The protocol consists of five components working in concert: a V2 Anchor smart contract that enforces allocation, locking, and LP policy immutably; a TrustScore engine that scores every token from 0–100 using on-chain and market data; a three-tier NFT ecosystem (Launch Certificates, Zodiac Badges, OG Pass — each serving a distinct purpose); a Creator Reputation system that tracks behaviour across all launches on-chain; and a TrustScore API with Chrome Extension that brings scoring into 9 trading platforms inline.

§ 2

The Problem — Solana's Trust Deficit

Solana processes tens of thousands of token launches every week. The vast majority follow the same pattern:

73%
LP Not Locked

Creator retains full control over pool liquidity and can remove it at any time

~60%
Hidden Allocations

Creator wallets hold undisclosed supply percentages enabling silent dumps

$2B+
Lost to Rug Pulls

Estimated annual losses across Solana DeFi from creator-controlled exits

Existing tools (rug checkers, LP scanners) are reactive — they report what already happened. What is missing is a protocol that makes rug pulls structurally impossible before launch.

Core insight

"The problem is not that creators are dishonest. The problem is that the infrastructure gives them the option to be. Remove the option — enforce rules in code — and the incentive to rug disappears."

§ 3

Solution — The HumbleTrust Protocol

HumbleTrust is not a launchpad in the traditional sense. It is a trust enforcement layer that any creator can opt into. By choosing HumbleTrust, a creator commits on-chain — not in text — to:

  • A fixed, publicly declared supply allocation across five immutable vaults
  • A creator allocation of 0–5% released in a time-locked vesting schedule (Day 30 / 60 / 90)
  • A liquidity policy: either LP tokens burned or locked in a program-derived account — never creator-controlled
  • A minimum lock period for any creator-held tokens (30–360 days)
  • An optional burn of locked tokens on unlock (0–100% of locked supply)
  • A transparent TrustScore calculated and published before launch

Every commitment is verified and enforced by the V2 Anchor smart contract. No admin key, no upgrade authority with special powers, no back door.

§ 4

V2 Smart Contract — Architecture

The V2 program is deployed at FGQ16c5cmDkmDRG27kt27VrZP3FnhHTH3qtrXoMg3PGr on Solana devnet. It is written in Rust using Anchor 0.32.1.

Five-Vault Supply Model

At launch, the creator declares the distribution of the total token supply across five vaults. All five must sum to exactly 100%. The contract validates this constraint before minting.

Bonding Curve Types

CPMM — Constant Product

Price follows x·y=k. Raydium-compatible. Standard AMM curve. Price rises predictably as supply is purchased. Optimal for DeFi utility tokens.

Quadratic Curve

Price = k·q². Slower early price increase. More community-friendly distribution. Early buyers get a larger advantage over later buyers.

Trade Fee Economics

A flat 1% fee is charged on every buy and sell. Split is immutable in the contract:

PLATFORM_FEE_BPS  = 50   // 0.5% → HumbleTrust treasury
CREATOR_FEE_BPS   = 50   // 0.5% → creator wallet (automatic, no claim needed)

LP Policy — Lock or Burn

At graduation, LP tokens are never sent to the creator wallet. The creator chooses one of two policies at launch time:

Lock Policy

LP tokens go to a program-derived lock PDA. Creator can claim LP swap fee share (40% standard / 60% premium) but cannot withdraw the LP itself until the lock expires.

Burn Policy

LP tokens are sent to the null address on graduation. Permanently removes any possibility of LP withdrawal. Maximum trust signal — indicated in TrustScore.

Raydium CPMM Graduation

When the bonding curve accumulates 50 SOL in the treasury, automatic migration to Raydium CPMM is triggered. Any wallet can call the migration instruction and receives 0.1 SOL as a migration reward.

  • SOL and token reserves move to Raydium CPMM pool via on-chain CPI
  • LP tokens routed to lock PDA or burn address per the declared LP policy
  • Creator begins earning LP fee share (40% or 60%) claimable monthly
  • Post-graduation trading continues on Raydium — no disruption

Anti-Bot Delay

Creators can set a delay of 0–600 seconds after launch before curve trading opens. During this window, buy instructions are rejected on-chain with error code TradingNotStarted (6027). Prevents sniper bots from front-running community buyers at launch.

Rug Risk Assessment

Independently of TrustScore, the engine produces a Rug Risk score (0–100) and categorises tokens as LOW / MEDIUM / HIGH / CRITICAL. Key flags include:

  • Mint authority active (+40 rug risk) — new supply can be printed at any time
  • Freeze authority active (+30) — creator can freeze any holder wallet
  • Honeypot pattern detected (+critical) — sell transactions blocked
  • LP less than 5% of market cap (+15) — shallow liquidity enables price manipulation
  • Top 10 holders >70% supply (+15) — extreme concentration risk

NFT Ecosystem — Three Distinct Types

Important

HumbleTrust has three entirely separate NFT types. Each serves a different purpose, has a different supply, a different price, and a different eligibility requirement. They are not interchangeable and should not be confused.

Type 1 · Automatic · Per Token Launch
Launch Certificate NFT (HTCERT)
A soulbound proof-of-launch credential minted automatically by the V2 program at the end of every successful HumbleTrust V2 token launch. It proves — on-chain and permanently — that the creator locked liquidity, declared a burn policy, and passed TrustScore validation.
Symbol: HTCERT Standard: Token-2022 NonTransferable Supply: Unlimited (1 per launch) Price: Free (gas only) Transferable: No (Soulbound)

Use case: Creators accumulate certificates across launches. TrustScore engine reads certificate history to compute the creator_track_record signal. High certificate count = verified launch history. Certificates cannot be transferred, sold, or revoked — they are permanent proof of conduct.

§ 7

Creator Reputation System

Each creator on HumbleTrust has a Creator Reputation PDA on-chain — a program-derived account that accumulates a verifiable history of their behaviour across all launches. This is separate from TrustScore (which scores individual tokens) and separate from NFTs.

Reputation Events

EventTypeEffect
Token Launch1Positive — records a new launch, increments launch counter
Token Unlock2Positive — records on-schedule unlock, confirms creator kept commitment
Community Complaint3Negative — deducts reputation, triggers review flag
Bonus Event4Positive — exceptional launch (high TrustScore, graduation) earns bonus

Events are recorded by the protocol's metrics_authority keypair via the record_reputation_event_v2 instruction. The authority signs server-side after validating the event — creators cannot self-report.

Reputation in TrustScore

A creator's reputation history feeds into the creator_track_record signal in the Legitimacy category of TrustScore. Verified issuers (creators with a sustained high reputation and multiple successful launches) earn the verified_issuer flag, which is shown on their tokens and displayed in the Discover page.

§ 11

Security Model

On-Chain Guarantees

  • PDA-based custody — all SOL and token vaults are program-derived accounts; no private key controls them
  • Mint authority revoked after launch — no additional supply can be minted
  • LP tokens go to lock PDA or burn address — creator cannot withdraw liquidity
  • Supply validation: all five vault percentages must sum to 100% (checked pre-mint)
  • Creator allocation hard cap: 5% enforced on-chain, not just in UI
  • Anti-bot delay: 0–600s before curve opens, configurable by creator at launch
  • Overflow-safe curve math: u128 intermediates prevent integer overflow in price calculations
  • Owner-check validation: SPL token accounts verified against spl_token::ID before CPI
  • LP fees constrained to pre-declared rewards_sol_wallet — arbitrary destination blocked

Infrastructure Security

  • API keys stored as SHA-256 hashes — plaintext never stored in database
  • Internal secrets compared via crypto.timingSafeEqual — timing-attack resistant
  • Rate limiting fail-closed — DB error blocks request, not allows it
  • Vercel serverless — no persistent server process to compromise
  • Supabase Row-Level Security on all tables
Devnet Alpha — Important Disclaimer

HumbleTrust is currently in devnet alpha. No real assets are at risk on the current deployment. Mainnet launch requires: (1) full security audit by a qualified Solana security firm, (2) Squads multisig upgrade authority migration, (3) production RPC with failover, (4) signed mainnet readiness checklist. Do not use real assets on devnet.

Reporting Vulnerabilities

Security researchers are encouraged to report findings via humble.trust@outlook.com or GitHub Security Advisories at github.com/HumbleTrust/humbletrust .
